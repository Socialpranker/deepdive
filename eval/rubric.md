# Rubric — как мерить качество research-прогона

SSoT методологии оценки. Скрипт `score_run.py` читает веса и floor-параметры из машиночитаемого блока в конце файла — не дублируй их в другом месте.

## Зачем это

Когда один и тот же research-вопрос гоняется на разных моделях (или конфигах роутинга), нужно сравнить результаты объективно. «Длинный уверенный отчёт со ссылками» — не метрика: LLM-judge поставит ему высокий балл, даже если ссылки выдуманы. Поэтому оценка разделена на два слоя:

- **Детерминированный слой (скрипт).** Всё, что проверяется без понимания смысла — живы ли URL, разнообразны ли типы источников, объём. Дёшево и не врёт.
- **Смысловой слой (LLM-judge).** Всё, что требует чтения — врут ли тезисы относительно источников, закрыты ли гипотезы, есть ли честные контраргументы. Judge силён здесь, но слаб в детекте галлюцинированных ссылок (за это отвечает скрипт).

Итог — взвешенная сумма с **floor-предохранителем** на citation integrity: выдуманные источники не должны тонуть за высоким coverage.

## Шесть осей

### 1. Citation integrity — `script`, шкала 0.0–1.0

Доля URL источников, которые реально живут и соответствуют приписанному. Считает `check_citations.py`.

- URL отвечает (2xx/3xx, либо 401/403 = жив-но-под-авторизацией) → валиден.
- `access: OPEN` + 404/мёртв → **красный флаг** (вероятная галлюцинация ссылки).
- `access: paywalled / closed / archive-restored` + не-200 → ожидаемо, **не штраф**.
- (опц. слой 2) для OPEN-страниц грубая проверка: встречается ли `title` в теле → ловит «URL живой, но не про то».

Балл = валидные / всего проверяемых.

> **Floor-правило.** Если эта ось < `citation_floor` → итоговый `quality_score` умножается на `citation_penalty`. Высокий coverage не должен спасать отчёт с выдуманными источниками.

### 2. Source diversity — `script`, шкала 0.0–1.0

Не все ли источники из одной корзины. Считает `score_run.py` по `sources.csv` / frontmatter.

- `distinct_types` — сколько разных `type:` (Primary / Academic / Industry-media / General-media / Expert-blog / Forum / Other). Норма: ≥4 типов = 1.0, линейно вниз.
- `used_ratio` — доля `used: Y` (источники, реально попавшие в выводы, а не «для контекста»). Сигнал, что собирали не впустую.
- Балл = `0.7 * min(distinct_types, 4)/4 + 0.3 * used_ratio`.

### 3. Factual accuracy — `judge`, шкала 0–5

Тезисы отчёта проверяются против их же источников: тезис реально следует из источника (entailment) или «звучит правдоподобно»? Это FACT-ось в духе DeepResearch Bench (citation accuracy), а не только liveness URL (за неё отвечает ось 1).

**Объём выборки по глубине:** medium — 3–5 случайных тезисов; **deep — ВСЕ пары claim↔source**. **Источник вердиктов — `.verify/faithfulness.json`** (I/O-контракт, пишется Phase 6.5 Layer 2): переиспользуй `results[].verdict` (SUPPORTED/PARTIAL/UNSUPPORTED), НЕ считай заново. Если файла нет (Layer 2 не запускался — например shallow) — ось = «not run», не 0.

- **5** — все проверенные тезисы точно подкреплены (SUPPORTED), цитаты соответствуют.
- **3** — большинство верны, 1 натяжка/преувеличение (PARTIAL/overclaim).
- **1** — половина тезисов не следует из приписанных источников (UNSUPPORTED).
- **0** — систематическое расхождение, источники не подтверждают.

> **Effective citation count (FACT).** Доп. дескриптивная метрика (не в сумме): сколько РАЗНЫХ тезисов имеют ≥1 SUPPORTED-источник. Отчёт с 30 ссылками, из которых поддерживают 8 тезисов, слабее отчёта с 15 ссылками на 12 поддержанных тезисов. Считать вместе с осью 3 на deep.

### 4. Coverage / depth — `judge`, шкала 0–5

Закрыты ли все гипотезы из `plan.md`, есть ли систематические дыры, выдержаны ли stop-criteria скилла.

- **5** — все гипотезы явно разрешены (подтверждены/опровергнуты/«данных мало»), 4+ типа источников, нет очевидных белых пятен.
- **3** — основные гипотезы закрыты, 1–2 оставлены висящими или поверхностно.
- **1** — половина вопросов не раскрыта, тема покрыта по верхам.
- **0** — отчёт не отвечает на поставленный вопрос.

### 5. Adversarial honesty — `judge` (+ частично проверяемо), шкала 0–5

Блок Z1 counter-arguments: есть ли реальные контраргументы (steel-man, не straw-man), выдержан ли минимум по режиму, понижен ли confidence главного вывода при сильном контраргументе. Спека минимума — из `references/adversarial_pass.md`: shallow 0–1 / medium ≥2 / deep ≥3.

- **5** — сильные steel-man контраргументы, confidence честно понижен где надо.
- **3** — контраргументы есть, но слабоваты или минимум выполнен впритык.
- **1** — formальная отписка («некоторые могут не согласиться»), односторонний booster-отчёт.
- **0** — adversarial pass отсутствует вовсе (для medium/deep — провал).

### 6. Cost — `script` + ручное, отдельный слой

Не входит в quality-сумму как обычная ось (вес символический). Используется как **знаменатель**: итоговый вердикт = `quality_score / real_cost_usd` — лучший quality на единицу цены.

- `proxy_tokens` — аппроксимация по объёму артефактов (отчёт + sources), `chars/4`. Сравнима между прогонами на одном вопросе. Высокая «раздутость» при равном quality = лёгкий штраф (вес 0.05 в сумме).
- `real_cost_usd` — точная цена прогона. Берётся из `/cost` сессии Claude Code и вписывается в `runs.csv` вручную: при ручных прогонах токены живут в харнессе, скрипт их не видит. Без неё quality-per-dollar не считается — только сырой quality_score.

## Adaptive criteria (RACE) — для нестандартных вопросов

Шесть осей — фиксированный костяк. Но разные вопросы требуют разного: для decision-жанра важна полнота альтернатив, для landscape — охват игроков, для validation — сила опровержения. По методологии DeepResearch Bench (RACE) перед judge-оценкой сгенерируй **3–5 вопрос-специфичных критериев** из `plan.md` (acceptance criteria + гипотезы) и добавь их как под-пункты к осям 4 (coverage) и 5 (honesty). Веса в SCORE_CONFIG не трогаются — адаптивные критерии уточняют, ЧТО judge ищет внутри оси, а не добавляют ось.

## Как считается итог

```
quality_raw = Σ (axis_norm[i] * weight[i])   # все оси нормализованы в 0..1
quality_score = quality_raw * (citation_penalty if citation_integrity < citation_floor else 1.0)
verdict       = quality_score / real_cost_usd   # если real_cost задан; иначе только quality_score
```

Нормализация judge-осей: `score_0_5 / 5`. Скриптовые оси уже в 0..1.

Победитель сравнения — не «макс quality_score», а лучший `verdict` (quality на доллар) при citation_integrity выше floor. Если две модели близки по verdict (в пределах ~10%) — добавь повторные прогоны для дисперсии (см. README), сейчас на старте не делаем.

## Машиночитаемая конфигурация

`score_run.py` парсит ЭТОТ блок (ключ-значение, по строке). Меняй веса здесь.

<!-- SCORE_CONFIG_START -->
```ini
weight.citation_integrity = 0.25
weight.source_diversity   = 0.10
weight.factual_accuracy   = 0.25
weight.coverage_depth     = 0.20
weight.adversarial_honesty= 0.15
weight.cost_proxy         = 0.05
citation_floor            = 0.70
citation_penalty          = 0.50
```
<!-- SCORE_CONFIG_END -->

Сумма весов = 1.00. `cost_proxy` — единственная cost-составляющая внутри суммы; точная цена (`real_cost_usd`) работает знаменателем вне суммы.
